# 프로젝트 목적
이 프로젝트는 대한민국 실거주 사용자를 위한 주택담보대출 상품 판정 웹앱을 만든다.

# 현재 단계
1단계 MVP는 "집 추천"이 아니라 "대출 가능 여부와 예상 한도 판정"이다.
사용자가 핵심 정보만 입력하면 아래를 빠르게 보여준다.
- 신생아 특례 디딤돌 가능 여부와 예상액
- 디딤돌 가능 여부와 예상액
- 보금자리론 가능 여부와 예상액
- 일반 주담대 예상 가능액
- 체증식 가능 여부
- 각 상품별 불가 사유
- 보유 현금을 합친 총 구매 가능 금액

# 제품 원칙
- 첫 화면은 입력이 적어야 한다.
- 사용자가 제일 궁금한 것은 "그래서 얼마까지 가능하냐"이다.
- 가능/불가만 보여주지 말고 이유와 예상 금액을 함께 보여준다.
- 정부대출과 일반대출을 분리해서 보여준다.
- 불가 사유는 반드시 사람이 이해 가능한 한국어 문장으로 보여준다.
- 상품별 기준은 하드코딩된 숫자 흩뿌리기 방식이 아니라 정책 룰 파일에 모아 관리한다.
- 정책 기준은 시행일 기준으로 바뀔 수 있으므로 룰 파일 분리 구조를 우선한다.

# 반드시 지원할 상품
- 신생아 특례 디딤돌
- 디딤돌
- 보금자리론
- 일반 주담대(보수적 추정)

- 배우자 연소득
- 보유 현금
- 기존 월 부채 상환액
- 무주택 여부
- 생애최초 여부
- 혼인 상태
- 직업 형태

- 자녀 수

- 지역 구분(수도권/비수도권, 규제지역 여부)

- 결혼비용
- 상세 입지 선호

# UX 원칙
- 결과 카드 최상단은 "예상 최종 대출 가능액"이다.
- 그 아래 상품별 카드가 나온다.
- 각 카드에는 상태, 예상액, 핵심 사유 1~3개가 들어간다.
- 상태는 가능 / 조건부 / 어려움 3단계로 보여준다.
- 숫자는 만원 단위 기준으로 입력하고 결과는 원화 보기 좋게 포맷한다.
- 숫자 입력은 기본값 0이 고정되지 않게 만든다.
- 비어 있는 입력은 빈 문자열 상태로 관리하고, 계산할 때만 숫자로 변환한다.

# 기술 원칙
- Next.js + TypeScript + Tailwind 사용
- 계산 로직은 lib/에 둔다.
- 타입은 types/에 둔다.
- 폼 상태와 계산 로직을 분리한다.
- 상품별 판정 함수는 나눠서 만든다.
- 하드코딩 숫자는 lib/policies/loanRules.ts 같은 정책 파일로 모은다.

# 결과 원칙
- 결과는 예상치다.
- 실제 가능 여부는 금융기관 심사, 최신 규제, 증빙서류에 따라 달라질 수 있음을 반드시 표시한다.
- 자영업자/프리랜서는 소득 입증 상태에 따라 달라질 수 있음을 별도 안내한다.
- 체증식은 나이, 소득형태, 상품 요건을 반영해 따로 판정한다.

# 작업 방식
- 한 번에 전체를 다 만들지 말고 단계별로 만든다.
- 먼저 폼 최소화
- 그 다음 상품별 판정 엔진
- 그 다음 결과 카드
- 마지막으로 문구와 포맷 정리
